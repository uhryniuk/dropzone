# Feature: Host Integration

## 1. Overview

How shimmed binaries reach the user's `PATH`. `~/.dropzone/bin/` is the directory added to `PATH`; `dz install` writes a POSIX wrapper script at `~/.dropzone/bin/<name>` that invokes the entrypoint from inside the package's bundled rootfs (see `binary_shimming.md`). Conflict detection keeps dropzone from silently overwriting system-installed binaries.

Works on both Linux and macOS. Shell-rc detection differs per-OS (bash/zsh on Linux; zsh default on macOS Catalina+, bash on older macOS, fish anywhere).

## 2. Goals

*   `~/.dropzone/bin` on the user's `PATH`, idempotently, on first run. Works for bash / zsh on both Linux and macOS.
*   Wrapper script written directly in `~/.dropzone/bin/<name>`. Because the wrapper points at `packages/<name>/current/rootfs/...`, `dz update` flips the `current` symlink and the wrapper continues to work unchanged.
*   Conflict policy that protects system-installed binaries (never overwritten) and predictably resolves dropzone-vs-dropzone clashes (overwrite with warning).
*   Clean unshim on `dz remove` and transparent re-target on `dz update`.

## 3. Components

### 3.1. `internal/hostintegration/path.go`

*   **`SetupDropzoneBinPath() error`**:
    *   Create `~/.dropzone/bin` if missing.
    *   Detect the active shell via `$SHELL`. Supported: bash, zsh. Anything else (fish, nushell, tcsh, …) → print the PATH export snippet to stderr and return without touching any rc file.
    *   For bash: edit `~/.bashrc` on Linux. On macOS, bash users may use `~/.bash_profile`, we append to both if they exist (or create `~/.bash_profile` if neither; macOS historically sources `~/.bash_profile` for login shells).
    *   For zsh: edit `~/.zshrc`. This is the default shell on macOS Catalina+ and common on Linux.
    *   Idempotent: look for an existing `export PATH=".../.dropzone/bin:$PATH"` line in a marked dropzone block. If present, no-op. If absent, append.
    *   Print a one-line nudge to `source` the rc file or open a new shell if the line was appended.

### 3.2. `internal/hostintegration/wrapper.go`

*   **`InstallWrapper(name string, content string) error`**, writes the wrapper script (content generated by `internal/shim/wrapper.go`) to `~/.dropzone/bin/<name>` with mode `0755`, subject to the conflict policy below.
*   **`RemoveWrapper(name string) error`**, removes `~/.dropzone/bin/<name>` if the file is a wrapper we wrote (starts with our shebang + a marker comment we include in generated wrappers). No-op with warning if it looks like something else took over the filename.

### 3.3. Conflict policy

On `InstallWrapper`:

1.  If `~/.dropzone/bin/<name>` does not exist: write, done.
2.  If it exists and looks like a dropzone-written wrapper (shebang + marker): overwrite with a warning. Typical on update or reinstall.
3.  If it exists as any other file (user-written script, unexpected symlink): skip with a warning naming the file. User resolves manually.

On `LookupPathConflict(name string)` before install:

1.  Check if `<name>` resolves on the user's `PATH` outside `~/.dropzone/bin/`. If so, print an informational note at install time: `"system <name> exists at /usr/bin/<name>; dropzone's shim will take precedence because ~/.dropzone/bin comes first on PATH."` Does not refuse the install.

## 4. CLI integration

No dedicated commands. Used by:

*   `App.Initialize()` → `SetupDropzoneBinPath()`
*   `packagehandler.InstallPackage` → `InstallWrapper`
*   `packagehandler.RemovePackage` → `RemoveWrapper`
*   `packagehandler.ApplyUpdate`, with digest-as-dir layout, flipping `current` is enough; the wrapper's content stays unchanged.

## 5. Technical details

*   **Wrapper not symlink:** we ship a POSIX shell script (generated in `internal/shim/wrapper.go`), not a symlink into the package dir. The wrapper references `packages/<name>/current` which transparently tracks the active digest across updates. See `binary_shimming.md` §4.4 for the script templates.
*   **Shell detection:** `filepath.Base(os.Getenv("SHELL"))`, matched against `bash` / `zsh`. Fish, nushell, tcsh, and other shells get the PATH snippet printed to stderr.
*   **RC file edits:** use `os.OpenFile` with `O_APPEND`. Lines are wrapped in a marked block for idempotency and easy removal:

    ```bash
    # >>> dropzone PATH >>>
    export PATH="$HOME/.dropzone/bin:$PATH"
    # <<< dropzone PATH <<<
    ```

    On rerun, we look for the `>>> dropzone PATH >>>` marker rather than string-matching the export line; makes re-detection robust against user reformatting.
*   **macOS shell file selection:** for bash, prefer `~/.bash_profile` (login shells) if it exists; fall back to `~/.bashrc`; create `~/.bash_profile` if neither exists. For zsh, `~/.zshrc`. For fish/nushell/others, print instructions only.
*   **Wrapper marker:** wrappers include a `# dropzone: <name> <version>` comment line after the shebang, used by `RemoveWrapper` to confirm the file belongs to dropzone before deleting.

## 6. Testing

### 6.1. Unit

*   `SetupDropzoneBinPath` idempotency on bash (linux + macOS rc-file selection), zsh, and unsupported-shell print path.
*   `InstallWrapper` against each conflict scenario (no existing, dropzone-written wrapper, user file, user symlink).
*   `RemoveWrapper` safety: doesn't delete files that don't match our marker.

### 6.2. Integration

*   Fresh home on Linux: `dz version` creates `~/.dropzone/bin` and appends the PATH block to `~/.bashrc` or `~/.zshrc` per `$SHELL`.
*   Fresh home on macOS: same, with `~/.bash_profile` preferred for bash users.
*   Install two packages whose entrypoints don't clash; both resolvable from the shell.
*   Install two packages with the same entrypoint name; second install overwrites with warning; first is no longer resolvable via dropzone.
*   Remove one; the surviving one still resolves. Remove the other; `~/.dropzone/bin/` is empty.

## 7. Open questions

*   **fish / nushell support.** Not MVP. Printed instructions are the fallback.
*   **`dz path` helper** that prints the PATH export snippet for users to paste into unsupported shells. Trivial, plausibly MVP.
*   **`dz doctor`** (mentioned in cli_foundations) could surface missing wrappers, orphan package dirs, and PATH not-set-up states.
